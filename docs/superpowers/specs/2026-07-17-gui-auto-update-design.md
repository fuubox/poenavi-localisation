# GUI 自動アップデート設計

## 目的

GitHub Release に新しい安定版が公開されたとき、Windows の exe 版利用者が、ぽえなびの GUI からダウンロード、検証、ファイル更新、再起動までを完了できるようにする。

更新対象は `v*` 形式のタグから作られた GitHub Release に限定する。`main` ブランチへの通常の push やプレリリース、ドラフトリリースは更新対象にしない。

## 現状と課題

- 起動時に `buri34/poenavi` の最新 Release を確認する処理は存在する。
- 新版を検出しても Release ページをブラウザで開くだけで、アプリ内から更新を適用できない。
- Windows では実行中の `PoENavi.exe` と同梱ファイルを自身で安全に置換できない。
- GitHub Actions がなく、リリース用ビルドと ZIP 添付は手作業になっている。
- `main.py` の `__version__` は `2.3.1` だが、現在の最新タグは `v2.4.0` であり、バージョン情報が同期していない。

## 採用方式

ぽえなび本体とは別の `PoENaviUpdater.exe` を同梱する。更新時はアップデーターを一時ディレクトリへコピーして起動し、本体終了後にインストール先を置換して再起動する。

PowerShell で生成した更新スクリプトは、実行ポリシーやセキュリティ製品の影響を受けやすいため採用しない。インストーラー基盤への全面移行は、現在の ZIP 配布方式に対して変更範囲が大きいため採用しない。

## リリース成果物

`vX.Y.Z` タグの push を契機に、GitHub Actions の Windows ランナーで次を実行する。

1. タグ名とコード上のバージョンが一致することを検証する。
2. テストを実行する。
3. PyInstaller で `PoENavi.exe` と `PoENaviUpdater.exe` をビルドする。
4. 配布ディレクトリを `PoENavi/` という単一のルートフォルダーで ZIP 化する。
5. `PoENavi.zip` と `PoENavi.zip.sha256` を生成する。
6. 同じタグの GitHub Release を作成し、両ファイルを添付する。

Release API から取得したリリースがドラフトまたはプレリリースの場合は無視する。自動更新が利用するアセット名は `PoENavi.zip` と `PoENavi.zip.sha256` に固定し、アセットが不足しているリリースは適用しない。

## コンポーネント

### バージョン情報

バージョン文字列を専用モジュールへ一元化し、GUI、更新判定、ビルド、タグ検証が同じ値を参照する。比較対象は SemVer の数値部分とし、先頭の `v` は入力時に正規化する。不正なタグは更新候補にしない。

### Release クライアント

GitHub Releases API との通信とレスポンス解析を GUI から分離する。タイムアウトを設定し、`User-Agent` を送信する。最新版のバージョン、リリースノート、ZIP URL、チェックサム URL、Release ページ URLを更新候補として返す。

### 更新マネージャー

本体プロセス内で次を担当する。

- 更新確認の開始と重複実行防止
- ZIP とチェックサムの一時領域へのダウンロード
- ダウンロード進捗とキャンセルの通知
- SHA-256 の一致確認
- ZIP の安全性検証
- 一時領域へコピーしたアップデーターの起動

ZIP 展開前に、絶対パス、`..`、シンボリックリンクなど、配布ルート外へ書き込めるエントリーを拒否する。期待する `PoENavi/PoENavi.exe` と更新マニフェストがない ZIP も拒否する。

### 専用アップデーター

アップデーターは本体 PID、本体のインストール先、検証済み ZIP、一時作業先を引数で受け取る。

1. 本体プロセスの終了を期限付きで待つ。
2. ZIP を一時ステージング領域へ展開する。
3. インストール先を同一親ディレクトリ内のバックアップ名へ変更する。
4. 新しい `PoENavi/` を元のインストール先へ移動する。
5. 新しい `PoENavi.exe` を起動する。
6. 起動に成功したらバックアップと一時ファイルを削除する。

置換または再起動に失敗した場合は、可能な限り新しいディレクトリを退避し、バックアップを元の名前へ戻す。復元結果と手動復旧に必要なバックアップ位置を日本語で表示する。

## ユーザーデータの扱い

`ConfigManager` が管理する設定、PoB データ、ラップ記録など、ユーザーデータ領域に保存されるファイルはインストール先の置換対象外である。

現在インストール先で編集可能な次のファイルは、更新時のデータ消失を避ける必要がある。

- `guide_data.json`
- `guide_data_poe2.json`
- `data/zone_data.json`

リリースには配布時ハッシュを記録した更新マニフェストを含める。現在のファイルが旧リリースの配布時ハッシュと同じなら未編集として新版へ置換し、異なる場合はユーザー編集済みとして新しいインストール先へ引き継ぐ。旧バージョンにマニフェストがない初回更新では、既知の編集可能ファイルを保守的に引き継ぐ。引き継いだファイルはバックアップにも残す。

## GUI と操作フロー

### 起動時確認

起動後にバックグラウンドで一度だけ確認する。新版がある場合は、バージョンとリリースノートを表示するダイアログを開く。

- `今すぐアップデート`: ダウンロード画面へ進む。
- `後で`: ダイアログを閉じる。

同じバージョンの通知を閉じた履歴は保持するが、手動確認では常に結果を表示する。

### 手動確認

既存の右クリックメニューに `アップデートを確認` を追加する。最新版なら「最新バージョンです」、通信失敗なら再試行可能なエラーを表示する。

### ダウンロードと適用

ダウンロード中はバージョン、進捗率、受信量、キャンセルボタンを表示する。検証完了後に更新開始の最終確認を出し、承認された場合だけアップデーターを起動して本体を終了する。

ソースコードから起動している場合は自動更新を行わず、Release ページを開く選択肢を表示する。

## エラー処理

- GitHub API のタイムアウト、接続失敗、API 制限は本体の起動を妨げない。
- 自動確認の失敗は通知を出さずに記録し、手動確認時だけ日本語で表示する。
- アセット不足、不正なバージョン、チェックサム不一致、不正 ZIP は更新を開始しない。
- ダウンロードのキャンセル時は一時ファイルを削除し、現在のアプリを継続する。
- インストール先へ書き込めない場合は本体終了前に検出し、管理者権限での自動再実行は行わない。
- 更新失敗時は現在版を復元し、復元できない場合もバックアップを削除しない。

## セキュリティ

- 通信先は GitHub API と、API が返す GitHub の Release アセット URLに限定する。
- HTTPS を使用し、リダイレクト後の配布元も GitHub 管理ドメインであることを検証する。
- SHA-256 が一致しない ZIP は展開しない。
- ZIP Slip とリンク経由のインストール先外への書き込みを拒否する。
- アップデーターへ渡すパスは絶対パスへ正規化し、対象インストール先に `PoENavi.exe` があることを確認する。
- シェル文字列を組み立てず、引数配列でプロセスを起動する。

## テスト方針

### 単体テスト

- 正常・同一・旧版・不正形式のバージョン比較
- Release API レスポンスの解析とプレリリース除外
- 必須アセットの選択
- チェックサム形式と SHA-256 検証
- 危険な ZIP エントリーの拒否
- 更新マニフェストによる編集済みファイル判定
- API、ダウンロード、キャンセル、検証失敗時の状態遷移

### 結合テスト

- 一時ディレクトリを模擬インストール先として、正常更新と再起動要求を確認
- 置換途中の失敗を注入し、旧版が復元されることを確認
- 編集済みガイドデータが保持され、未編集データは新版へ更新されることを確認
- スペースおよび日本語を含むインストールパスで確認

実プロセス終了と Windows GUI 再起動は、自動テストに加えてリリース前の手動スモークテストを行う。

## 完了条件

- `v*` タグから GitHub Release と更新用アセットが自動生成される。
- exe 版の GUI から更新確認、ダウンロード、検証、適用、再起動が完了する。
- 更新失敗時に旧版が起動可能な状態へ復元される。
- ユーザー設定と編集済みガイドデータが失われない。
- ソース実行、オフライン、壊れたリリースのいずれでも通常起動を妨げない。
- 更新関連の単体・結合テストが通り、既存テストに回帰がない。

## 対象外

- `main` の各 push を配布する開発版更新
- macOS、Linux 向け自動更新
- 差分パッチ配布
- コード署名証明書の導入
- アプリ内でのリリース作成やタグ作成
